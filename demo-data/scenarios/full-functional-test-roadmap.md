# Маршрутная карта полного тестового прогона

## Цель

Подготовить demo-набор данных для сквозной проверки уже реализованного функционала:

- импорт оргструктуры;
- импорт пользователей и кадровое обновление из Excel;
- создание и публикация учебного контента;
- наполнение тем материалами разных форматов;
- формирование тестового банка;
- прохождение обучения в обязательном контуре;
- прохождение тестов в самостоятельном режиме;
- проверка пользовательских и менеджерских экранов после появления результатов.

## Prerequisites

Перед Этапом 1 должны быть подтверждены следующие условия:

- demo DB поднята с чистой Flyway schema;
- backend запущен в `dev` profile;
- demo-admin существует;
- personnel create-user import либо реализован, либо выбран fallback bootstrap;
- content authoring/publication endpoints проверены;
- assignment campaign launch проверен на минимальном наборе;
- self-catalog/self-test endpoint доступен;
- analytics rebuild strategy известна.

## Рабочие ограничения текущего контура

Для полного e2e-прогона через текущий UI и runtime безопасно опираться на такие типы:

- материалы: `TEXT`, `PDF`, `DOCX`, `VIDEO`;
- вопросы для прохождения через learner UI: `SINGLE_CHOICE`, `MULTIPLE_CHOICE`.

Важно:

- текущий runtime материала хранит только `material record`-метаданные: `topicId`, `name`, `description`, `materialType`, `sortOrder`;
- в текущем content API и learner UI не подтверждён полноценный file upload / file storage / URL rendering контур;
- поэтому в demo-roadmap корректно говорить не «загрузить материалы как файлы в storage», а «создать material records и проверить, как backend/UI хранят и показывают metadata-записи»;
- для обязательного назначения курс должен быть `PUBLISHED`;
- каждая тема курса должна быть `PUBLISHED`;
- у каждой темы должен быть активный итоговый тест `CONTROL`, назначаемый как `FINAL_TOPIC_CONTROL`;
- self-contour должен использовать опубликованные self-visible тесты, которые не являются `active final test` темы;
- банк вопросов должен быть больше состава итоговых тестов, чтобы можно было вручную собирать разные тесты и проверять аналитику на разных вопросах;
- автоматическую рандомизацию состава вопросов нельзя закладывать в сценарий, пока она явно не подтверждена runtime-реализацией.

## Целевая структура demo-контура

Рекомендуемый объём для полного тестового прогона:

- `3` курса;
- `8` тем;
- `24-28` материалов;
- `11-13` тестов;
- `70-90` вопросов в общем банке.

Такой объём достаточно большой, чтобы проверить связность данных, но ещё остаётся управляемым для ручного демо.

## Demo-акторы

Для реального прогона нужно заранее зафиксировать пользователей-носителей сценариев:

| Actor | Назначение в прогоне |
|---|---|
| `ADMIN` | создаёт оргструктуру, пользователей, контент, публикует, запускает кампании |
| `EXPERT` | создаёт курсы, темы, materials records, вопросы и тесты |
| `OPERATOR-1` | проходит обязательный курс успешно |
| `OPERATOR-2` | проходит обязательный курс с ошибками |
| `MANAGER-1` | проверяет supervision и managerial analytics по своим сотрудникам |
| `LEARNER-SELF-1` | проходит self-test по промышленной безопасности на ОПО |

Примечание:

- `ADMIN` может быть отдельным demo-admin вне enterprise roster;
- тогда ожидаемое число пользователей в БД после bootstrap — `52`: `51` сотрудник из demo-пакета + `1` demo-admin.

## Набор курсов

### 1. Курс «Промышленная безопасность оператора НПЗ»

Цель:
проверить обязательное назначение для массового контура операторов.

Темы:

1. Вводный инструктаж по промышленной безопасности
2. Безопасная работа на технологической установке
3. Действия при нештатной ситуации

Материалы по темам:

- `TEXT`: краткий конспект и контрольные тезисы;
- `PDF`: локальный регламент или инструкция;
- `DOCX`: чек-лист допуска к смене;
- `VIDEO`: короткий учебный ролик или запись инструктажа.

Тесты:

- по 1 итоговому тесту на каждую тему;
- по 1 тренировочному тесту минимум для двух тем;
- в каждом итоговом тесте `8-10` вопросов;
- в банке по теме `14-18` вопросов.

Кому назначать:

- операторам `OPS` контура;
- в первую очередь на подразделения `CCK-UKK`, `CCK-UPV`, `CCK-MTBEO`.

### 2. Курс «Наряд-допуск и безопасная организация работ»

Цель:
проверить смешанный контур: обязательное назначение для ИТР и просмотр аналитики руководителем.

Темы:

1. Подготовка наряда-допуска
2. Контроль рисков перед началом работ
3. Роли руководителя и ответственного исполнителя

Материалы по темам:

- `TEXT`: методические заметки;
- `PDF`: шаблон наряда и пример заполнения;
- `DOCX`: памятка согласования;
- `VIDEO`: разбор типовой ситуации.

Тесты:

- по 1 итоговому тесту на каждую тему;
- по 1 дополнительному тренировочному тесту для тем 1 и 3;
- в каждом итоговом тесте `7-9` вопросов;
- в банке по теме `12-15` вопросов.

Кому назначать:

- начальникам установок;
- начальникам комплексов;
- специалисту по ОТиПБ;
- при необходимости части ИТР `HEAD` контура.

### 3. Курс «Самоподготовка по промышленной безопасности на ОПО»

Цель:
проверить self-service каталог, самостоятельное прохождение и историю результатов в предметной области ОПО без mandatory assignment.

Темы:

1. Основные требования промышленной безопасности на ОПО
2. Типовые нарушения и действия работника при угрозе аварии

Материалы по темам:

- `TEXT`: памятка по базовым требованиям и запретам на ОПО;
- `PDF`: локальная памятка по промышленной безопасности;
- `DOCX`: рабочая памятка по безопасному поведению и эскалации;
- `VIDEO`: короткий ролик с примерами правильных и ошибочных действий работника.

Тесты:

- по 1 self-visible тесту на каждую тему;
- 1 общий самостоятельный тест по всему курсу;
- в каждом тесте `6-8` вопросов;
- в банке по теме `10-12` вопросов.

Кому открывать:

- всем пользователям learner-контура как self-testing каталог;
- без обязательного назначения на первом прогоне.

Архитектурное правило:

- self-тесты должны быть `PUBLISHED` и доступны в self-contour;
- self-тесты не должны использоваться как `FINAL_TOPIC_CONTROL` для mandatory assignment;
- если тема участвует и в assigned, и в self-контуре, для неё нужно разводить отдельный final-control тест и отдельный self-visible тест.

## Правила наполнения банка вопросов

Для каждой темы подготовить:

- не менее `60%` вопросов `SINGLE_CHOICE`;
- не менее `40%` вопросов `MULTIPLE_CHOICE`;
- минимум `2-3` вопроса с правдоподобными дистракторами;
- минимум `1-2` вопроса, где правильных вариантов несколько;
- минимум `1` пограничный вопрос, который полезен для аналитики ошибок.

Рекомендуемая схема:

- в банке темы `14` вопросов;
- в итоговый тест попадает `8` вопросов;
- избыточные вопросы остаются в банке и могут использоваться для ручной сборки альтернативных тестов.

Это позволит проверить:

- что банк шире состава итоговых тестов;
- что можно собрать несколько тестов по одной теме без расширения content scope;
- что аналитика по вопросам и темам накапливается на разных наборах вопросов.

## Сценарии полного прогона

### Этап 1. Базовая загрузка demo-контура

Последовательность:

1. Импортировать оргструктуру.
2. Импортировать пользователей и привязки.
3. Проверить baseline workbook для users/personnel.

Ожидаемый результат:

- в системе есть оргдерево НПЗ;
- загружены `51` demo-пользователь;
- доступны роли operator, manager, expert и профильный ИТР-контур.

Измеримые признаки успеха:

- `organizational_unit = 13` по demo-org contour;
- `app_user = 52`, если demo-admin создаётся отдельно от enterprise roster;
- `app_user = 51`, если проверяется только enterprise roster без отдельного demo-admin;
- `user_organization_assignment > 0`;
- `user_role_assignment > 0`.

### Этап 2. Создание и публикация контента

Последовательность:

1. Создать 3 курса.
2. Создать 8 тем и связать их с курсами.
3. Для каждой темы создать material records минимум в 3-4 форматах, без расширения scope до file-storage, если upload/runtime не подтверждён.
4. Создать банки вопросов.
5. Создать итоговые тесты по темам.
6. Для части тем создать тренировочные тесты.
7. Перевести контент в `PUBLISHED`.

Ожидаемый результат:

- каждый курс открывается в каталоге;
- у каждой темы есть материалы;
- у каждой темы есть валидный итоговый тест для назначения.

Измеримые признаки успеха:

- `course = 3`;
- `topic = 8`;
- `material = 24-28`;
- `question = 70-90`;
- `test = 11-13`;
- у каждой темы есть `active final control test`.

### Этап 3. Проверка обязательного назначения

Последовательность:

1. Запустить assignment campaign на операторский курс для выбранного подразделения.
2. Запустить assignment campaign на курс по наряду-допуску для managerial-контура.
3. Проверить создание assignment и assignment tests.
4. Проверить карточку назначенного обучения у получателей.

Ожидаемый результат:

- курс появляется в разделе назначенного обучения;
- внутри назначения видны темы, материалы и список тестов;
- по каждой теме создан `FINAL_TOPIC_CONTROL`.

Измеримые признаки успеха:

- `assignment_campaign >= 2`;
- `assignment > 0`;
- `assignment_test > 0`.

### Этап 4. Прохождение обязательного назначения

Проверяемые роли:

- оператор с активным назначением;
- начальник установки с активным назначением;
- руководитель, который затем смотрит аналитику.

Последовательность:

1. Открыть назначенное обучение.
2. Просмотреть материалы по всем темам.
3. Пройти минимум один тест успешно.
4. Пройти один тест с ошибками.
5. Повторить прохождение на другом пользователе для накопления статистики.

Ожидаемый результат:

- появляются assigned attempts и результаты;
- статус назначения меняется предсказуемо;
- по темам и пользователям накапливаются данные для аналитики.

Измеримые признаки успеха:

- `test_attempt > 0`;
- `result > 0`;
- assigned attempts и self attempts не смешаны по `attempt_mode`;
- для assigned results заполнены assignment-related поля, для self results они пусты.

### Этап 5. Проверка самостоятельного режима

Последовательность:

1. Открыть learner catalog.
2. Найти self-course по промышленной безопасности на ОПО.
3. Запустить self-test без назначения.
4. Завершить одну попытку успешно.
5. Завершить вторую попытку с другим набором ответов.
6. Проверить историю self-results.

Ожидаемый результат:

- self-testing доступен без assignment;
- попытки и результаты сохраняются отдельно от assigned-контура;
- в истории видны несколько самостоятельных результатов.

Измеримые признаки успеха:

- `test_attempt` содержит строки с `attempt_mode = SELF`;
- `result` содержит строки с `attempt_mode = SELF`;
- self results не содержат `assignment_id` и `assignment_test_id`.

### Этап 6. Проверка менеджерских экранов и итоговой связности

Последовательность:

1. Под менеджером открыть текущий supervision.
2. Проверить аналитику по сотрудникам.
3. Проверить аналитику по подразделениям.
4. Если агрегаты не пересчитываются автоматически, выполнить предусмотренный rebuild/recalculation step.
5. Сверить темы, проценты прохождения и проблемные вопросы.

Ожидаемый результат:

- менеджер видит сотрудников своего контура;
- агрегаты по темам не пустые;
- есть различие между успешно и неуспешно пройденными тестами;
- результаты self и assigned не смешиваются некорректно.

Checkpoint:

- если analytics rebuild запускается автоматически, это нужно зафиксировать перед прогоном;
- если rebuild не автоматический, перед managerial analytics нужен явный технический шаг пересчёта;
- если rebuild strategy ещё не реализована, это должно быть честно отражено в acceptance-checkpoints.

Измеримые признаки успеха:

- analytics aggregates не пустые, либо явно зафиксировано, что rebuild не запускался;
- supervision-экран показывает сотрудников managerial scope;
- user-topic и department-topic analytics возвращают непустые строки после корректного rebuild шага.

## Минимальный состав файлов, который нужно подготовить дальше

В `demo-data` имеет смысл добавить следующие артефакты:

- описание курсов и тем;
- таблицу состава материалов по каждой теме;
- таблицу тестов и связи тестов с темами;
- таблицу банка вопросов;
- краткий сценарий ручного прогона по ролям.

Рекомендуемое размещение:

- `demo-data/scenarios/full-functional-test-roadmap.md` — эта маршрутная карта;
- `demo-data/scenarios/content-matrix.md` — матрица курсы/темы/материалы/тесты;
- `demo-data/scenarios/acceptance-checkpoints.md` — SQL/API checks после каждого этапа;
- `demo-data/scenarios/role-runbook.md` — пошаговый прогон по ролям;
- `demo-data/templates/` — шаблоны Excel/CSV, если появится отдельный импортный контур для учебного контента.

## Приоритет следующего шага

Следующим практическим шагом нужно подготовить `content-matrix.md`, где по каждой теме будет зафиксировано:

- имя курса;
- имя темы;
- какие именно 4 материала загружаются;
- какие тесты создаются;
- сколько вопросов лежит в банке;
- сколько вопросов попадает в итоговый тест;
- в каком контуре используется тема: `ASSIGNED`, `SELF` или оба.

После этого можно переходить к фактической подготовке файлов материалов и текстов вопросов.
